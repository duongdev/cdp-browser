# Git workflow

The smallest workflow that keeps history readable. Solo project today; assume a contributor could join tomorrow.

## Branches

- `main` — always green, always shippable.
- Feature branches off `main`, merged back via PR (even solo, for the audit trail).

### Branch naming

```
<type>/t<NNN>-<short-kebab-name>     # when the work is a docs/tasks/ task
<type>/<short-kebab-name>            # when there is no task (ad-hoc chore/fix)
```

Types:
- `feat/` — new capability
- `fix/` — bug fix
- `refactor/` — pure restructure, no behavior change
- `docs/` — documentation only
- `chore/` — config, deps, tooling
- `test/` — test additions or improvements
- `spike/` — throwaway exploration (never merges to main)

Examples: `feat/t003-adaptive-viewport`, `fix/t007-letterbox-offset`, `refactor/extract-notification-dedup`, `docs/update-adr-0003`, `chore/biome-config`, `spike/ime-input-prototype`.

**Repo-internal task IDs vs. private PM IDs — know the difference:**

- The `tNNN` prefix is the **task ID** from `docs/tasks/NNN-*.md`. It lives in the repo, is open-source-safe, and *should* be stamped into the branch so the work traces back to its spec.
- **Never** include private PM-tool references (Linear/Jira/Notion ticket IDs). That metadata is confidential and stays out of the repo entirely — branches, commits, PRs, code, comments.

## Commit messages

**Semantic title only. Lowercase. No body.**

```
<type>(<scope>): <description> (tNNN)
```

The ` (tNNN)` suffix is the task ID — repo-internal, open-source-safe, makes every commit traceable to its spec. Omit it only when the change is genuinely not tied to a task (ad-hoc chore). This is the *only* sanctioned ID in a commit title; private PM IDs never appear.

Types: `feat`, `fix`, `refactor`, `docs`, `chore`, `test`, `perf`, `style`, `build`, `ci`, `revert`.

Scope: the module or area affected — `viewport`, `tabs`, `notifications`, `adaptive-viewport`, `settings`, `main`, `renderer`, etc. Omit for repo-wide changes.

Description: imperative present tense, lowercase, no trailing period.

**Good:**
- `feat(notifications): add teams toast capture via side-channel (t005)`
- `fix(viewport): correct letterbox offset when dpr > 1 (t008)`
- `refactor(tabs): extract closed-tab stack to pure function`
- `docs(adr): add adr-0003 on notifications side-channel`
- `chore: bump electron to 34`
- `test(adaptive-viewport): add reduce state machine coverage (t011)`

**Bad:**
- `Fix bug` — too vague, capitalized, no scope
- `feat(viewport): Add letterbox fix` — capitalized
- `wip: stuff` — never
- `feat(notifications): add teams capture and fix letterbox and update settings` — multi-thing

If you need to explain *why* a change was made, that goes in the **PR description**, not the commit body. Commit titles stay one line.

## Atomic commits

Each commit does one thing and leaves the system green (lint + types + relevant tests pass). See [code-quality.md](code-quality.md) for the principle.

Squash merges *into main* are fine when a feature branch has many WIP-style commits — the squash commit message follows the rules above.

Prefer **rebase + push --force-with-lease** on feature branches over merge commits for cleanup. Never rebase shared branches without coordination.

---

## Hard rules

These come from project-wide constraints and override everything else.

1. **Never auto-commit.** Commit only when the user says "commit".
2. **Never auto-push.** Push only when the *current* message says "push" — prior approvals don't carry forward.
3. **Never attribute work to AI.** No `Co-authored-by: Claude`, no "generated by", no AI references in commits, code, comments, branch names, or PRs.
4. **Never reference private PM IDs** in code, commits, branches, or PRs.
5. **Never `--no-verify` or skip hooks** unless the user explicitly asks.

## Pull requests

Even solo, every change to main goes through a PR. Cheap to do; gives a self-review beat and an audit trail.

### PR template

```markdown
## What

One paragraph. The change in plain language.

## Why

The motivation — link to the task file or ADR if relevant.

## How

Brief technical summary. Modules touched. Contract changes if any.

## Tests

- [ ] Unit tests added/updated (pure logic)
- [ ] Manual smoke test performed (CDP/IPC surface, visual review)
- [ ] Visual review via Chrome MCP (if UI-touching)
- [ ] All quality gates green (pnpm check, pnpm typecheck, pnpm test)

## Docs

- [ ] CLAUDE.md updated for affected modules
- [ ] ADR written (if architectural decision made)
- [ ] Task closed (status done, moved to docs/tasks/done/)

## Out of scope

What this PR explicitly does not do. Future work captured as new tasks.
```

### Self-review pass

Before opening a PR (even solo):

1. Re-read your own diff, top to bottom, with fresh eyes.
2. Run `pnpm check`, `pnpm typecheck`, `pnpm test` one more time.
3. Verify the doc updates match what you actually changed.
4. Manually exercise the feature with a live Remote Browser.
5. Then open the PR.

The cost of a 10-minute self-review is much less than the cost of merging a regression.
