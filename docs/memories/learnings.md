**2026-05-29** â€” A clicked Electron **OS notification** silently did nothing (banner showed, click â†’ no tab focus, no deep-open) because the `Notification` was a local `const` with no retained reference: V8 garbage-collects the JS `Notification` object, and a collected Notification never delivers its `click` event. Fix: hold every shown Notification in a module-level `Set` (`liveNotifications` in `main.js`) until it is clicked or closed. The diagnosis method that nailed it after several wrong guesses: fire a **synthetic `cdp:notification-activate` from main straight into the renderer** (a temporary env-gated hook), bypassing the OS click â€” the remote Teams conversation switched, proving the renderer / `chromeSend` / `openTeamsThread` path was fine and isolating the failure to OS-click delivery. Lesson: when an Electron notification click "does nothing", suspect Notification GC first; and when a multi-stage handler fails, trigger the downstream stage directly to bisect renderer-vs-transport-vs-OS rather than reasoning about the whole chain.

**2026-05-29** â€” Changing the notification entry shape (t028 added `activate`/`groupKey`/`adapter`) silently broke Teams/Outlook **Web Push** deep-open while the in-app path kept working. Root cause: three *hand-picked field lists* along the push chain each re-projected the entry to the pre-t028 schema and were never extended â€” the server push payload (`web/server.mjs` `sendPushToAll`), the service worker's `options.data` (`public/sw.js`), and the renderer's serviceWorker-`message` entry reconstruction (`cdp-web-transport.ts`). TypeScript can't catch this: the chain crosses process boundaries (server â†’ SW â†’ page) as untyped JSON, and `CdpNotification` even allowed the new fields as optional, so dropping them type-checked clean. Lesson: when a cross-boundary payload shape changes, grep for every manual `{ id, targetId, â€¦ }` re-projection of it â€” the SSE/in-app path that forwards the *whole* entry will mask the regression on the path that doesn't. Guarded now by a regression test asserting the SW-message path preserves `activate`.

**2026-05-29** â€” The default-OFF direct-frame tunnel (`/api/cdp-ws/{tabId}`) was removed when the Downlink seam landed (t021). The original motivation was to pipe raw CDP frames from a second dedicated WS socket straight to the browser, bypassing SSE. In practice this was slower on WebKit/iPad for two reasons: (1) a second CDP session competes with the server's main screencast socket for frame production â€” the remote browser serialises delivery between multiple clients on the same target, so adding a second listener reduces throughput for both; (2) forwarding raw CDP JSON meant the browser had to decode a `Page.screencastFrame` envelope and base64-decode the image data itself, which WebKit's JS engine handles significantly slower than the binary `Blob â†’ createObjectURL` path the SSE/WS dispatcher already uses. Keeping the Screencast Frame to a single source (through the one Dispatcher) eliminated both costs and removed ~85 lines of de-dup branching. Do not re-add it: if lower-latency frame delivery is ever needed, profile first â€” the bottleneck is almost always the remote browser's frame production rate, not the delivery path.
